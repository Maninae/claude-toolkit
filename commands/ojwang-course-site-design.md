# Skill: Course Site Design System

Design system and UX conventions for building course lecture websites. This skill captures the component library, interaction patterns, and design decisions — not hardcoded colors. The palette is swappable via CSS custom properties; the patterns and structure are what matter.

Use the `teach` skill alongside this one: `teach` governs **writing quality**, this skill governs **visual presentation and UX**.

---

## Architecture

A static HTML site with no build step:
- `site/index.html` — landing page with module cards
- `site/lectures/lectureNN.html` — one file per lecture
- `site/styles/shared.css` — single shared stylesheet, CSS custom properties for theming
- `site/scripts/shared.js` — sidebar TOC generation, KaTeX rendering, scroll progress, mobile nav

No frameworks. No bundler. Plain HTML + CSS + vanilla JS. KaTeX loaded from CDN for math rendering.

---

## Design Token Architecture

All colors, fonts, and spacing are defined as CSS custom properties in `:root`. To retheme the site, swap the variable values — never hardcode colors in component rules.

### Token categories

- **Backgrounds:** `--bg-deep`, `--bg-surface`, `--bg-elevated`, `--bg-hover` — four layers from darkest to lightest
- **Text:** `--text-primary`, `--text-secondary`, `--text-muted` — warm tones, never pure white on dark
- **Accents:** `--accent` (primary, used for links/tags/numbers), `--teal` (secondary, used for keywords/labels/progress)
- **Semantic colors:** `--blue` (definitions), `--green` (insights), `--purple` (theorems), `--amber` (examples) — used for callout borders, backgrounds, and syntax highlighting
- **Rules:** `--rule`, `--rule-strong` — subtle dividers at two opacity levels
- **Fonts:** `--serif` (headings), `--sans` (body), `--mono` (code, labels, tags)

### Palette is variable

The design system works with any palette that follows this structure. When changing colors, maintain the relationships:
- Background layers must have enough contrast between them
- Text colors should be warm and readable against the deepest background
- Accent colors need to pop against `--bg-surface`
- Semantic colors should be visually distinct from each other

## Typography Hierarchy

| Level | Font | Size | Usage |
|-------|------|------|-------|
| Hero title | `--serif` | 48px | Landing page title |
| Hero hook | `--sans` | 22px, bold | Provocative opening question (accent color) |
| Lecture h2 | `--serif` | 32px | Section headings |
| Lecture h3 | `--serif` | 24px | Subsection headings |
| Lecture h4 | `--sans` | 18px, semibold | Minor headings |
| Body | `--sans` | 17px | Paragraphs, lists |
| Roadmap | `--sans` | 18px | Opening paragraph, slightly larger |
| Eyebrow | `--mono` | 12px, uppercase | "Lecture N", "Course Notes" |
| Tags | `--mono` | 12px, uppercase | Topic pills |
| Callout labels | `--mono` | 11px, uppercase | "Definition", "Key Insight" |
| Code/pseudocode | `--mono` | 14px | All code content |

### Key rules

- **No KaTeX math in headings.** The sidebar TOC is auto-generated from `<h2>` and `<h3>` text. KaTeX renders as oversized formatted math in the sidebar. Use HTML entities instead: `&alpha;` for α, `&#772;` for combining overline, `<sub>` for subscripts.
- **No italic serif at large sizes.** Bold sans for prominent display text — italic serif hurts readability at hero/header scale.
- **Headings use sentence case**, not title case.

## Page Structure

### Landing page (`index.html`)

```
hero (eyebrow + title + hook question + stats)
  └─ module sections (numbered, with lecture cards)
       └─ lecture cards (title + description + tags)
```

- Hero includes a **provocative question** as a hook (`.hero-hook`, bold, accent color)
- Module sections have large semi-transparent numbers and title blocks
- Lecture cards have hover lift animation and accent left-border reveal
- Prereq cards in a grid (3-col desktop, 2-col tablet, 1-col mobile)

### Lecture pages

```
sidebar (course title + auto-generated TOC)
progress bar (gradient, full width)
lecture header (eyebrow + title + description + tags)
top nav (compact prev/next text links)
article
  └─ roadmap paragraph
  └─ sections (h2/h3 hierarchy)
  └─ callouts, algorithms, code blocks, pseudocode
bottom nav (prev/next cards)
```

## Navigation Patterns

### Sidebar TOC
- Auto-generated by JS from `<h2>` and `<h3>` elements in `.lecture-article`
- `h2` → `.nav-item`, `h3` → `.nav-sub-item` (indented)
- Scroll-based active tracking (120px threshold)
- Mobile: hidden off-screen, toggled by hamburger menu

### Top navigation (`.lecture-nav-top`)
- Compact, monospace, 13px
- `← Previous Title` on left, `Next Title →` on right
- Uses `&#8592;` and `&#8594;` for arrows
- Placed between `</header>` and `<article>`
- First lecture: next link only (auto-aligns right). Last lecture: next points to assessment.

### Bottom navigation (`.lecture-nav`)
- Full card-style with "Previous" / "Next" labels and lecture titles
- Cards have hover lift and border glow
- First lecture uses `<span></span>` spacer for flex layout

### Consistency rule
Top and bottom nav must have identical prev/next hrefs and titles. When adding or reordering lectures, update both.

## Component Library

### Callout boxes (5 types)

```html
<div class="callout definition|theorem|example|insight|note">
  <div class="callout-label">Label Text</div>
  <p>Content...</p>
</div>
```

| Type | Color | When to use |
|------|-------|-------------|
| `definition` | Blue | Formal definitions, mathematical statements |
| `theorem` | Purple | Core results, proofs, important equations |
| `example` | Amber | Worked examples, concrete instances |
| `insight` | Green | Key takeaways, "aha" moments, intuition |
| `note` | Muted | Asides, "looking ahead" previews, caveats |
| `prereq` | Rose (`--rose`) | "This Lecture Is For You If" — 2-4 bullets helping readers decide to read or skip. Placed after roadmap, before first h2. |

### Algorithm blocks

```html
<div class="algorithm">
  <div class="algorithm-title">Algorithm N: Name</div>
  <ol>
    <li><span class="keyword">repeat</span> until converged:</li>
    <li style="padding-left: 2em;">$x_0 \sim \text{dataset}$ <span class="comment">// description</span></li>
  </ol>
</div>
```

Formal numbered algorithms with KaTeX math. Use for canonical algorithms (DDPM Algorithm 1, DDIM sampling, etc.).

### Code blocks (real code)

```html
<div class="code-block">
  <div class="code-block-label">PyTorch</div>
  <span class="keyword">import</span> torch<br>
  ...
</div>
```

For actual runnable code (PyTorch, Python). Has a language label in the top-right corner. Syntax highlighting via span classes.

### Pseudocode blocks

```html
<div class="pseudocode">
  <span class="keyword">def</span> <span class="function">train_step</span>(model, x_0, schedule):<br>
  ...
</div>
```

For illustrative pseudocode. Distinguished from real code by teal left border (vs. full border on code blocks). Same syntax highlighting token classes. Optional `<div class="pseudocode-title">Title</div>` for labeled blocks. Some blocks in lectures 04-05 use `<pre>` inside — spans work fine inside `<pre>`.

### Syntax highlighting tokens

These classes work in `.code-block`, `.pseudocode`, and `.algorithm`:

| Class | Color variable | Applies to |
|-------|---------------|------------|
| `.keyword` | `--teal` | `def`, `return`, `for`, `if`, `class`, `import`, `while`, `None`, `True`, `False` |
| `.function` | `--blue` | Function/method names at call or definition sites |
| `.number` | `--amber` | Numeric literals |
| `.string` | `--green` | Quoted strings |
| `.comment` | `--text-muted` (italic) | Comments after `#` or `//` |
| `.type` | `--purple` | Type names when used as types |

**Don't over-highlight.** Variable names, operators, and most identifiers stay plain. Only tag the categories above.

### Collapsible sections

```html
<details class="collapsible">
  <summary>Section title</summary>
  <div class="collapsible-content">
    ...
  </div>
</details>
```

Use for optional deep-dives (full ELBO derivation, extended proofs) that would break the main narrative flow.

### Math hover tooltips

Add hover explanations to specific terms in KaTeX equations using `\htmlData`:

```latex
$$L = \left\| \varepsilon - \htmlData{tip=The neural network predicting the noise}{
  \varepsilon_\theta(x_t, t)} \right\|^2$$
```

This renders normally but adds a dotted teal underline. On hover, a dark popup appears with the plain-language explanation.

**Requirements:**
- KaTeX config must have `trust: true` (set in `shared.js` `renderMath()`)
- CSS class `.math-tooltip` handles the popup (JS-driven, appended to `<body>` to escape overflow clipping)
- `[data-tip]` selector styles the dotted underline and cursor

**When to use:**
- Opaque notation that would benefit from a plain-English gloss but doesn't deserve its own paragraph, subsection, or callout
- Terms that combine multiple symbols into one concept (e.g., `ε_θ(...)` = "the neural network predicting noise")
- Brief elaborations that help the reader without breaking the flow — a tooltip is lighter than a sentence of prose
- Use generously throughout equations. If a reader might pause and think "what does this part mean?", it's a good candidate
- **Every Definition callout with a complex formula should have at least one tooltip** on the most opaque term

**Where to place tooltips — the geometric mean rule:**

Don't tooltip the simplest terms (the student already knows those) or the outermost structure (the surrounding text explains that). Instead, target the **geometric mean of difficulty** — the mid-complexity subexpression that, once understood, unlocks the rest of the formula.

Think of a formula as a tree of nested expressions, each with a "parsing difficulty level" for a student. If the formula is `Level2(Level2, Level5(Level3, Level3))`, the best tooltip target is the `Level5(...)` subexpression — it's the bottleneck where comprehension stalls.

**Example:** In $L_{\text{simple}} = \mathbb{E}\left[\left\| \varepsilon - \varepsilon_\theta(\sqrt{\bar{\alpha}_t}\, x_0 + \sqrt{1-\bar{\alpha}_t}\, \varepsilon, t) \right\|^2\right]$:
- The outer $\mathbb{E}[\cdot]$ and $\|\cdot\|^2$ are simple (expectation, squared norm) — no tooltip needed
- $\varepsilon$ alone is simple (the noise we sampled) — no tooltip needed
- $\varepsilon_\theta(\ldots, t)$ is the geometric mean: "the neural network (with weights θ) predicting what noise was added" — **good tooltip target**
- The argument $\sqrt{\bar{\alpha}_t}\, x_0 + \sqrt{1-\bar{\alpha}_t}\, \varepsilon$ could get a second tooltip: "the noisy image at timestep t, constructed from the clean image and sampled noise"

**Syntax:** `\htmlData{tip=Your tooltip text here}{LaTeX content to underline}`

**Limitations — tip text must be plain ASCII:**
- No `%` (LaTeX comment character — use "percent" instead)
- No `()` parentheses (break KaTeX parsing — use commas or dashes to rephrase)
- No unicode characters like `θ`, `—`, `α` (use plain ASCII: "theta", "alpha", commas instead of em-dashes)
- No nested `\htmlData` (make tooltips siblings, not parent-child)
- No LaTeX commands in tip text — plain ASCII English only

## Responsive Design

Three tiers:

| Breakpoint | Layout |
|------------|--------|
| Desktop (> 960px) | Sidebar visible, 2-col module grid, full typography scale |
| Tablet (769-960px) | Sidebar visible, module grid collapses to 1-col |
| Mobile (≤ 768px) | Sidebar hidden (hamburger toggle), full-width content, reduced type sizes, vertical nav |

Key mobile changes:
- Hero h2: 48px → 32px
- Article h2: 32px → 26px, h3: 24px → 20px
- All padding shrinks (60px → 24px horizontal)
- Bottom nav stacks vertically
- Top nav font shrinks to 12px

## Design Decisions Log

Decisions made during development, with rationale:

1. **Top + bottom nav:** Users shouldn't have to scroll to navigate between lectures. Top nav is compact (just text links), bottom nav is prominent (full cards). Both exist on every lecture.

2. **No KaTeX in headings:** Sidebar TOC auto-generates from heading text. KaTeX renders at full display size in the sidebar, breaking layout. Use HTML entities (`&alpha;`, `&#772;`, `<sub>`) for Greek letters and subscripts in headings.

3. **Pseudocode gets syntax highlighting:** Same token classes as real code blocks, just with a different container style (teal left border vs. full border). Makes pseudocode more scannable.

4. **Code block line-height: 1.45:** Was 1.7/1.8 (too spacious, felt double-spaced). Tightened to 1.45 for both `.code-block` and `.pseudocode`.

5. **Roadmap intros on every lecture:** A 2-4 sentence paragraph (`.lecture-roadmap`) immediately after `<article>` opens, previewing the lecture's journey. Not meta-commentary ("In this lecture we will...") but a motivating preview.

6. **Catchy hooks on landing page:** The hero section leads with a provocative question before any descriptive text.

7. **Bold sans for display text:** No italic serif at large sizes — hurts readability.

8. **Reparameterization trick placement:** Demoted from full section to "Looking Ahead" callout in the forward process lecture. Training mechanics belong in the training lecture.

9. **Noise schedules before SNR:** Students need concrete schedule examples before the formal comparison framework.

10. **Math hover tooltips:** Complex KaTeX terms get `\htmlData{tip=...}` hover explanations. Uses JS-based tooltip appended to `<body>` (not CSS `::after`) because `.katex-display` has `overflow-y: hidden` that clips CSS-only tooltips. Use generously for opaque notation.

11. **All inline math in KaTeX:** Any math symbol in prose (x₀, αₜ, ε, N(0,I)) must be wrapped in `$...$`. Bare ASCII math in running text is a rendering bug.

12. **No premature implementation details:** Don't mention optimizer, learning rate, normalization range, or batch size before the lecture that teaches them. Each detail appears first where it's properly introduced.

13. **Em-dash discipline:** Prefer colons (for explanations), commas (for related clauses), or periods (for dramatic pauses that are really two sentences). Reserve em-dashes for genuine parenthetical insertions.

14. **Section numbers on all h2 headings:** Every `<h2>` inside `<article>` gets a `<span class="section-number">X.Y</span>` prefix (e.g., "3.5"). Orange monospace, 0.75em, 0.7 opacity. Only h2s, not h3 subsections. Header h2 outside article is not numbered.

## Checklist: Adding a New Lecture

1. Copy an existing lecture HTML as template
2. Update: title, eyebrow number, description, tags
3. Add both top nav (`.lecture-nav-top`) and bottom nav (`.lecture-nav`)
4. Update prev/next links in the adjacent lectures (both top and bottom nav)
5. Add a `.lecture-roadmap` paragraph after `<article>` opens
6. Ensure no KaTeX math in `<h2>` or `<h3>` tags — use HTML entities
7. Use appropriate callout types for definitions, theorems, insights
8. Syntax-highlight all pseudocode and code blocks with span classes
9. Update `index.html` module data if adding to a new module
10. Test mobile layout (hamburger menu, stacked nav)
